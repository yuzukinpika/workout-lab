# 筋トレ Lab — 開発引き継ぎドキュメント

Claude Code で開発を継続するための仕様・設計・意思決定・ロードマップのまとめ。
**正となるコードは `workout-app.html`（単一ファイル）** です。

---

## 1. 概要

- 名称: 筋トレ Lab
- 形態: **単一HTMLファイルのWebアプリ**（スマホのホーム画面に追加して使う想定）。ビルド不要・外部ライブラリ不使用・ネット不要。
- 保存: **ブラウザの localStorage**（端末内のみ）。バックアップ/読み込みはJSONで手動エクスポート。
- 技術: 素の HTML / CSS / Vanilla JS。グラフは自前のインラインSVG（Chart.js等は未使用）。
- UI: モバイル前提（`max-width:560px`）、**ライトテーマ（暖色ホワイト×オレンジ/レッドのアクセント）**。下部タブナビ。

---

## 2. 画面構成

タブは3つ：**記録 / 履歴 / 分析**。モーダルは3つ：**種目ピッカー / 記録編集 / エクスポート設定シート**。

- **記録**: 日付（タップ式カレンダー、デフォルト今日）→ 種目（ライブラリから選択）→ セット入力（重量×回数/RPE）。前回値を自動表示。片側種目は「両側/片側」を選択。保存すると「この日の記録」に追加。
- **履歴**: 日付別に全記録を一覧。タップで編集モーダル（セット修正・日付変更・削除）。
- **分析**: 習慣化（週連続達成ストリーク・週目標リング・バッジ）／活動カレンダー（月表示・トレした日を塗る）／ボリューム推移（週・月・任意期間）／部位バランス（今週）／種目別の詳細（推定1RM・ボリューム折れ線）／RPE推移／トレーニング頻度／体重推移／設定／AIコーチ用エクスポート／データ管理。

---

## 3. データモデル（localStorage）

- キー: `workoutLab_v4`（旧バージョン `_v3`/`_v2` からは `load()`→`migrate()` で自動移行）

```
DB = {
  entries: [ {
    id: string,
    date: "YYYY-MM-DD",
    exName: string,          // 種目名
    part: "胸|背中|肩|腕|脚|体幹",
    equip: "barbell|dumbbell|machine|cable|bodyweight|assist|other",
    uni: boolean,            // 片側ずつ行う種目か
    sides: 1 | 2,            // uni種目の実施範囲（両側=2）。非uniは1
    sets: [ { w:number, reps:number, rpe:number|null } ]
  } ],
  customEx: [ { name, part, equip, uni, custom:true } ],  // 自作種目
  favs: [ exName ],          // お気に入り
  archived: [ exName ],      // アーカイブ済み
  weekGoal: number,          // 週間目標（回/週）既定4
  bwLog: [ { date:"YYYY-MM-DD", kg:number } ],  // 体重ログ（日付付き）
  profile: { goals:[string], weak:string, injury:string, days:string }, // AIエクスポート用
  balanceMetric: "sets" | "volume",  // 部位バランスの指標
  balanceMin: number,        // 目安レンジ下限（既定10）
  balanceMax: number,        // 目安レンジ上限（既定20）
  rmFormula: "average" | "epley" | "brzycki"
}
```

組み込み種目リストは定数 `BUILTIN`（`[名前, 部位, 器具, 片側ずつ?]`）。補助筋マップは削除済み（下記の「マイキャラ」参照）。

## 4. 重要なロジック（仕様として維持すべき点）

### ボリューム計算 `setVolume(equip,w,reps,uni,sides,bw)`
- バーベル/マシン/ケーブル/その他: `w * reps`
- ダンベル: **片手の重量を入力**し `w * 2 * reps`（両手同時種目の慣習）
- 片側ずつ種目(uni): `w * reps * sides`（両側=×2、片側のみ=×1）
- 自重(bodyweight): `(体重 + 加重w) * reps`
- アシスト(懸垂補助など): `(体重 - 補助w) * reps`
- `bw`（体重）は **`getBW(date)`** が bwLog から「その日以前で最も新しい記録」を返す（過去の自重種目も当時の体重で計算）。

### 推定1RM `est1RM(w,reps)`（`DB.rmFormula` で切替）
- Epley: `w*(1+reps/30)`
- Brzycki: `w*36/(37-reps)`
- average（既定・推奨）: EpleyとBrzyckiの平均
- 注記: どの式も近似。低レップほど正確、10〜12回超で誤差大。1回=そのまま。

### 週の定義
- `weekKey(date)` は**月曜始まり**の週開始日(YYYY-MM-DD)を返す。

### セキュリティ（XSS対策・維持必須）
- **出力エスケープ**: `innerHTML` に入れるユーザー由来の文字列（種目名・ID・部位・セット値など）は必ず `esc()` を通す。新しい描画コードを書くときも同様。
- **JS文字列への埋め込み禁止**: `onclick="fn('${name}')"` の形はエスケープしても破れる（HTML属性内でデコードされるため）。種目ピッカーは `__pickItems` 配列＋`pickAct(index, action)` のインデックス参照、履歴カード等は `data-*` 属性＋`this.dataset.x` を使う。
- **入力サニタイズ**: `migrate()` が読み込みデータ（localStorage・バックアップJSON）を信頼せず型強制する。数値は `Number` 化、日付は `YYYY-MM-DD` 形式チェック、`equip`/`part`/`balanceMetric`/`rmFormula` は既知の値へホワイトリスト。スキーマ変更時もこの方針を守る。
- **CSP**: `<meta http-equiv="Content-Security-Policy">` で外部への接続・画像読み込み等を全て遮断（`default-src 'none'`）。万一注入されてもデータを外部送信できない。外部リソースを使う機能を足すときはここの緩和が必要。

### ストリーク（連続記録）`calcStreak()`
- **「週目標（`weekGoal`）を達成した週が何週連続で続いているか」**を返す（単位は週）。
- 毎日連続の日数カウントは廃止（休養日が必要な筋トレの実態に合わないため）。
- 今週はまだ途中でも達成済みならカウントし、未達成でもストリークは切らない（先週から遡って数える）。
- バッジも週ベース（3週連続達成・8週連続達成）。

### 部位バランス
- 指標は `balanceMetric`（セット数 or ボリューム）で切替。セット数モードのみ `balanceMin/Max` で不足/適正/過多を判定。

## 5. AIコーチ用エクスポート（費用ゼロのAI連携）

「設定シート」で **目標(複数選択)・弱点・ケガ・週の可能日数・出力期間(直近2週〜1年/全期間/任意)・対象種目** を指定 → `buildExport()` が **Markdownファイル** を生成しダウンロード。中身：

1. AIへの依頼プロンプト（「まず分析→対話で一緒に決める」流れ。次回メニューを一気に決めさせない）
2. 私の状況（目標・弱点・ケガ・可能日数・体重※最終記録日付き）
3. 基本情報（期間・総セッション・週連続達成・ボリューム定義）
4. **アプリの自動観察メモ**（`observations()` が停滞種目・部位過不足・RPE高止まり・頻度低下・体重トレンドを自動検出）
5. 週間ボリューム/頻度/平均RPE/体重の推移（数値）
6. 種目別の推移（推定1RM・ボリューム・各回のセット）

→ これをチャット版AI（無料）に添付して個別アドバイスを得る設計。将来 Claude API を組み込めばアプリ内で直接提案も可能。

## 6. これまでの意思決定（背景）

- **プラットフォーム**: App Store等を避け、まず単一HTML(≒PWA的)で最短実装。
- **AI**: コスト回避のため「ルールベース → ファイルエクスポートで外部AIに渡す」を採用（API連携は将来オプション）。
- **ダンベル**: 片手の重量を入力し自動で×2（一般的なジム表記に一致）。
- **片側種目**: 記録時に両側/片側を選択（既定=両側）。自作種目もON/OFF可。
- **1RM**: 「唯一正しい式」は無いため、Epley＋Brzyckiの平均を既定にしつつ選択可。
- **体重**: 単一値ではなく日付付きログにし、過去ボリューム計算に反映。
- **削除した機能**:
  - スケジュール/予定タブ（固定グリッドは価値が低く、将来「次回セッション計画」として再設計する想定）
  - マイキャラ（部位がボリュームで成長するSVGキャラ。見た目が不十分で撤去。関連の補助筋マップ`SECONDARY`も削除）
  - 活動ヒートマップ（GitHub芝型・直近18週。週3〜4回のトレだと歯抜けに見えて達成感につながらないため、月カレンダー表示に置換）
  - 日数ベースのストリーク（休養日で途切れて逆インセンティブになるため、週目標の連続達成に変更）
- **デザイン**: ダークテーマ（紺×水色）→ ライトテーマ（暖色ホワイト×オレンジ/レッド）に変更（2026-07、明るさと筋トレらしさを優先）。

## 7. ロードマップ（次の候補）

- 習慣化の強化: リマインダー通知、達成アニメーション、目標設定UIの充実
- 記録の時短: インターバルタイマー、ルーティン（テンプレ）呼び出し、前回セットのワンタップ複製の強化
- スケジュール再設計: 提案エンジンが組む「次回セッション計画」→記録画面へ流し込み、記録→分析→計画のループを閉じる
- Claude API 連携: アプリ内で個別メニュー提案・自然文コーチング（従量課金）
- PWA化: `manifest.json` + Service Worker でインストール/オフライン対応
- 種目メタ拡充: より細かい筋群マッピング、動画/フォーム解説リンク

## 8. Claude Code での進め方メモ

- `workout-app.html` を起点に。まずブラウザで開いて「分析→デモデータを入れて試す」で全機能を確認。
- 現状は単一ファイル。規模拡大なら (a) このまま増築、(b) HTML/CSS/JSを分割、(c) Vite+React等へ移行、のいずれか。移行する場合も **上記のデータモデルと計算ロジックは仕様として維持** してください。
- `migrate()` と localStorage スキーマ（キー `workoutLab_v4`）は後方互換の要。変更時はバージョンを上げて移行関数を追加。
- グラフは自前SVG（`lineSVG` / バー / 月カレンダー）。ライブラリ導入は任意。
- テストのしやすさ: 純粋関数（`setVolume` `est1RM` `weekKey` `getBW` `observations` `buildExport`）は DOM 非依存で単体テスト可能。
